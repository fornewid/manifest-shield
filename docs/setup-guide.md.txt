# Manifest Shield — AI Agent Setup Guide

This guide helps AI agents (Claude Code, GitHub Copilot, Gemini, Cursor, etc.)
correctly install and configure the Manifest Shield Gradle plugin.

## What is Manifest Shield?

A Gradle plugin that detects unintentional changes to Android's merged
`AndroidManifest.xml`. It saves a baseline and fails the build when the manifest
changes — catching unexpected permissions, components, or features added by
dependency updates.

- Plugin ID: `io.github.fornewid.manifest-shield`
- Maven Central: `io.github.fornewid.manifest-shield:manifest-shield`
- Minimum: AGP 8.0.0+, Gradle 8.0+

## Step 1: Apply the plugin

Add to the **module's** `build.gradle.kts` (not the root project):

Using Version Catalog (`libs.versions.toml`):

    [plugins]
    manifest-shield = { id = "io.github.fornewid.manifest-shield", version = "<latest-version>" }

    // module/build.gradle.kts
    plugins {
        id("com.android.application")
        alias(libs.plugins.manifest.shield)
    }

    manifestShield {
        configuration("release")
    }

Or directly:

    plugins {
        id("com.android.application")
        id("io.github.fornewid.manifest-shield") version "<latest-version>"
    }

    manifestShield {
        configuration("release")
    }

**AI Agent**: The minimal setup is just `configuration("release")` with no
additional properties. All security-relevant categories are enabled by default.
Do NOT add `sources = true`, `metaData = true`, or any other properties unless
the user explicitly requests them. Do NOT apply this plugin in the root
`build.gradle.kts` — it must be applied to each Android module individually.

## Step 2: Generate a baseline

    ./gradlew manifestShieldBaseline

This creates `manifestShield/releaseAndroidManifest.txt` in the module directory.
This file should be committed to version control.

## Step 3: Detect changes

    ./gradlew manifestShield

If the merged manifest differs from the baseline, the build fails with a diff.
To accept the changes, re-run `./gradlew manifestShieldBaseline`.

The `manifestShield` task is automatically included in `./gradlew check`, so
CI pipelines running `check` will also verify the manifest baseline.

**AI Agent**: Always run `manifestShieldBaseline` first during initial setup.
Running `manifestShield` before a baseline exists will create one automatically,
but the recommended flow is to generate and review the baseline explicitly.

## Default behavior

The plugin guards the following by default (no configuration needed):

- `<uses-permission>`
- `<uses-feature>` (only `android:required="true"` entries)
- `<activity>`, `<activity-alias>`, `<service>`, `<receiver>`, `<provider>` (only `android:exported="true"` components)
- `android:permission` attributes on exported components

**AI Agent**: These are ALREADY enabled. You MUST NOT include them in the
configuration block. Do not write `usesPermission = true`, `activity = true`,
or any other default-true property — not even as "explicit for clarity." If the
user asks to "track permissions and activities," the correct answer is that they
are tracked by default and no configuration change is needed.

## Optional features (opt-in)

These are disabled by default. Only add them when the user requests:

| Property | Description |
|----------|-------------|
| `sources = true` | Generate source-attributed baseline grouped by library/module |
| `queries = true` | Track `<queries>` (package visibility declarations) |
| `metaData = true` | Track `<meta-data>` entries |
| `intentFilter = true` | Track `<intent-filter>` on components |
| `usesSdk = true` | Track `<uses-sdk>` (minSdkVersion, targetSdkVersion) |
| `permission = true` | Track `<permission>` declarations |
| `usesPermissionSdk23 = true` | Track `<uses-permission-sdk-23>` |
| `usesLibrary = true` | Track `<uses-library>` (only `android:required="true"` by default) |
| `usesNativeLibrary = true` | Track `<uses-native-library>` |
| `profileable = true` | Track `<profileable>` |
| `startup = true` | Track `androidx.startup` initializer declarations |

Example with opt-in features:

    manifestShield {
        configuration("release") {
            sources = true
            metaData = true
        }
    }

## Important notes

- `exportedOnly = true` (default) only includes components with explicit
  `android:exported="true"`. On Android 12+, all components with intent-filters
  must set this attribute — components without it are excluded from the baseline.
- Intent-filters are never merged between manifests by AGP. Each dependency's
  intent-filters are added independently. Enable `intentFilter = true` to track
  these changes.

## Filtering options

These options are `true` by default and control what appears in the baseline:

| Property | Default | Description |
|----------|---------|-------------|
| `exportedOnly` | `true` | Only include components with `android:exported="true"` |
| `requiredOnly` | `true` | Only include `<uses-feature>` and `<uses-library>` with `android:required="true"` |

**AI Agent**: Do NOT set `exportedOnly = false` or `requiredOnly = false` unless
the user explicitly asks to include all components or all features regardless of
their exported/required status.

## Configuration property names

Use the exact Kotlin property names listed below. Do NOT use abbreviations or
alternative names.

| Correct | Wrong |
|---------|-------|
| `usesSdk` | `sdk` |
| `usesPermission` | `permissions` |
| `usesFeature` | `features` |
| `activity` | `activities` |
| `activityAlias` | `activity-alias`, `activityAliases` |
| `service` | `services` |
| `receiver` | `receivers` |
| `provider` | `providers` |

## Common mistakes to avoid

**AI Agent**: Pay special attention to these:

1. Do NOT add properties to `configuration()` unless the user asks. The defaults
   are carefully chosen for security-focused, low-noise baselines. Even if the
   user says "track permissions," respond that it is already tracked — do NOT
   add `usesPermission = true`.
2. Do NOT use `sources = true` by default. It generates an additional
   `*.sources.txt` file that most projects don't need.
3. Do NOT set `exportedOnly = false` or `requiredOnly = false` by default.
   These exist for users who want to track all entries, not just security-relevant ones.
4. Do NOT confuse `usesPermission` (guards `<uses-permission>` tags) with
   `permission` (guards `<permission>` declaration tags).
5. The baseline file (`manifestShield/releaseAndroidManifest.txt`) should be
   committed to version control.
6. The plugin only supports `com.android.application` modules. Library modules
   produce incomplete baselines (missing transitive dependency manifests) due to
   AGP's manifest merge design. Do NOT apply to library or dynamic-feature modules.
7. Do NOT apply the plugin in the root `build.gradle.kts`.

## Not tracked

The following manifest elements are not tracked by this plugin:

- `<permission-group>`, `<permission-tree>` — rarely used
- `<instrumentation>` — test-only
- `<property>` — Android 12+ element, may be supported in a future version
